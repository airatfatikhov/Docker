Для копирования собранного JAR-файла на удаленный хост в GitLab CI/CD лучше всего использовать протокол SCP (Secure Copy) поверх SSH. Это стандартный, безопасный и надежный способ.
Ниже приведено пошаговое руководство с подробным объяснением каждого этапа.

# 📋 Шаг 1: Подготовка удаленного хоста

Перед настройкой пайплайна убедитесь, что на целевом сервере (куда будем копировать) выполнены следующие условия:
SSH-сервер запущен (sshd).
Существует пользователь, под которым будет происходить деплой (например, deployer или app-user).
У этого пользователя есть права на запись в директорию, куда будет сохранен JAR (например, /opt/myapp/).

# 🔑 Шаг 2: Настройка аутентификации (SSH Keys)

GitLab Runner должен иметь возможность подключиться к серверу без ввода пароля. Для этого используем SSH-ключи.
Вариант А: Использование приватного ключа (Рекомендуемый для простоты)
Сгенерируйте пару ключей на вашей машине (или на сервере):

````
ssh-keygen -t ed25519 -C "gitlab-ci-deploy" -f deploy_key
````

Публичный ключ (deploy_key.pub) добавьте на удаленный хост в файл ~/.ssh/authorized_keys пользователя, под которым идет деплой.
Приватный ключ (deploy_key) скопируйте целиком.
В GitLab перейдите в Settings → CI/CD → Variables.
Добавьте новую переменную:
Key: SSH_PRIVATE_KEY
Value: Вставьте содержимое файла deploy_key (начинается с -----BEGIN OPENSSH PRIVATE KEY-----).
Flags: Отметьте Masked (чтобы скрыть в логах) и Protected (если нужно).
Type: File (опционально, но лучше просто Variable, мы запишем его в файл в скрипте).
Вариант Б: Использование пароля (Менее безопасно, не рекомендуется)
Если вы не можете использовать ключи, можно использовать sshpass, но это требует установки дополнительного ПО и менее безопасно. Мы рассмотрим вариант с ключами.

# ⚙️ Шаг 3: Настройка .gitlab-ci.yml

````
stages:
  - build
  - deploy

# ... ваши предыдущие джобы сборки ...

deploy-to-server:
  stage: deploy
  image: alpine:latest  # Легкий образ, содержащий ssh-client и scp
  before_script:
    # 1. Устанавливаем необходимые утилиты (openssh-client содержит ssh и scp)
    - apk add --no-cache openssh-client
    
    # 2. Создаем директорию для SSH ключей
    - mkdir -p ~/.ssh
    
    # 3. Записываем приватный ключ из переменной в файл
    # Важно: chmod 600 обязателен, иначе SSH откажется использовать ключ
    - echo "$SSH_PRIVATE_KEY" > ~/.ssh/id_ed25519
    - chmod 600 ~/.ssh/id_ed25519
    
    # 4. (Опционально) Отключаем строгую проверку хоста, если вы используете IP или динамические DNS
    # В продакшене лучше добавить отпечаток хоста в known_hosts заранее
    - echo "StrictHostKeyChecking no" >> ~/.ssh/config
    - echo "UserKnownHostsFile /dev/null" >> ~/.ssh/config

  script:
    # 5. Копируем JAR файл на удаленный сервер
    # Синтаксис: scp [опции] локальный_файл пользователь@хост:удаленная_директория
    - scp build/libs/*.jar ${DEPLOY_USER}@${DEPLOY_HOST}:${DEPLOY_PATH}/app.jar

    # 6. (Опционально) Перезапуск приложения на удаленном сервере
    # Используем ssh для выполнения команды на удаленной машине
    - ssh ${DEPLOY_USER}@${DEPLOY_HOST} "sudo systemctl restart myapp-service"
    
    # Или, если нет systemd, просто запуск java:
    # - ssh ${DEPLOY_USER}@${DEPLOY_HOST} "cd ${DEPLOY_PATH} && nohup java -jar app.jar > app.log 2>&1 &"

  rules:
    - if: $CI_COMMIT_BRANCH == "main"  # Деплоим только из main
      when: manual  # Ручной запуск для безопасности (можно убрать when: manual для авто-деплоя)

  environment:
    name: production
    url: http://${DEPLOY_HOST}:8080
````

# 📝 Шаг 4: Добавление переменных окружения в GitLab
DEPLOY_HOST - 	
IP или домен удаленного сервера
DEPLOY_USER - Пользователь на сервере
DEPLOY_PATH - Директория, куда копировать JAR
SSH_PRIVATE_KEY - Приватный SSH ключ (см. Шаг 2)

# 🔍 Подробное объяснение каждого шага скрипта
image: alpine:latest:
Мы используем минимальный Linux-образ. Он маленький и быстрый.
По умолчанию в Alpine нет ssh и scp, поэтому их нужно установить.
apk add --no-cache openssh-client:
Устанавливает клиентские утилиты SSH. Флаг --no-cache уменьшает размер образа после установки.
mkdir -p ~/.ssh:
Создает стандартную директорию для конфигурации SSH.
echo "$SSH_PRIVATE_KEY" > ~/.ssh/id_ed25519:
Берет значение переменной SSH_PRIVATE_KEY из GitLab и записывает его в файл.
SSH-клиент ожидает найти ключ в этом файле (или указанном явно).
chmod 600 ~/.ssh/id_ed25519:
Критически важный шаг! SSH отказывается работать с ключами, которые имеют права доступа для чтения другими пользователями (group/other). Права 600 означают: чтение и запись только для владельца.
StrictHostKeyChecking no:
При первом подключении к новому серверу SSH спрашивает: "Do you trust this host? (yes/no)". В CI/CD мы не можем ответить "yes" интерактивно.
Эта настройка автоматически принимает любой хост.
⚠️ Риск: Это делает вас уязвимым к атакам Man-in-the-Middle.
✅ Лучшая практика: Сгенерировать отпечаток хоста (ssh-keyscan -H $DEPLOY_HOST) и добавить его в ~/.ssh/known_hosts в разделе before_script.
scp build/libs/*.jar ...:
build/libs/*.jar: Путь к артефакту внутри контейнера GitLab Runner. Gradle обычно кладет JAR сюда.
${DEPLOY_USER}@${DEPLOY_HOST}: Куда подключаемся.
${DEPLOY_PATH}/app.jar: Куда сохраняем. Мы переименовываем файл в app.jar для удобства, чтобы имя не менялось от версии к версии.
ssh ... "systemctl restart ...":
После копирования файла нужно перезапустить приложение, чтобы оно подхватило новую версию.
Команда выполняется на удаленном сервере.

# 🛡️ Безопасность и лучшие практики
Не храните ключи в репозитории! Только в GitLab Variables.
Используйте отдельного пользователя. Не давайте права root пользователю deployer. Если нужно перезапускать сервис, настройте sudoers:

````
# На удаленном сервере в /etc/sudoers.d/deployer
deployer ALL=(ALL) NOPASSWD: /bin/systemctl restart myapp-service
````

# 🚀 Альтернатива: rsync
Если файлов много или нужно синхронизировать директории, вместо scp можно использовать rsync (тоже входит в пакет rsync в Alpine):

````
- apk add --no-cache openssh-client rsync
- rsync -avz -e "ssh -o StrictHostKeyChecking=no" build/libs/*.jar ${DEPLOY_USER}@${DEPLOY_HOST}:${DEPLOY_PATH}/
````

rsync быстрее при повторных деплоях, так как передает только изменения, но для одного JAR-файла разница незаметна. scp проще и понятнее.
